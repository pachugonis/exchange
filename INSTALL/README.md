# Установка ExchangeKit на VPS

Установка одним скриптом по модели **лицензии**: клиент получает лицензионный
ключ после оплаты, установщик активирует его на лицензионном сервере, скачивает
**подписанный артефакт релиза** и разворачивает приложение. git-доступ к
исходникам клиенту не выдаётся.

Скрипт [install.sh](install.sh) разворачивает нативно (без Docker):

- бэкенд — самодостаточный бандл `server.mjs` как systemd-сервис `exchangekit-api` (порт `4000`);
- фронтенд (`dist/`) раздаётся через nginx, обращения к `/api` проксируются на бэкенд;
- таблицы БД и админ-аккаунт создаёт сам сервер при первом запуске.

> Для оператора (вас): как собирать и публиковать релизы и поднять лицензионный
> сервер — см. [../LICENSE/README.md](../LICENSE/README.md) и
> [release.sh](release.sh). Этот документ — про установку у клиента.

---

## 1. Требования

| Что | Значение |
|-----|----------|
| ОС | Ubuntu 24.04 LTS (чистый сервер) |
| RAM | от 1 ГБ (рекомендуется 2 ГБ) |
| Диск | от 10 ГБ |
| Доступ | root (или пользователь с `sudo`) |
| Домен | A-запись домена должна указывать на IP сервера |
| Лицензия | лицензионный ключ `LIC-XXXX-XXXX-XXXX-XXXX` и e-mail, на который он оформлен |

Сервер должен иметь доступ в интернет к лицензионному серверу (по умолчанию
`https://license.exchangekit.cc`) — оттуда скачивается релиз.

---

## 2. Что нужно для установки

Вы получаете после оплаты:

1. файл `install.sh` (этот установщик);
2. **лицензионный ключ** и **e-mail**, на который оформлена лицензия.

Скопируйте `install.sh` на сервер (например через `scp`) или вставьте в файл:

```bash
scp install.sh root@IP_СЕРВЕРА:/root/
```

git-репозиторий клонировать не нужно — код приедет внутри подписанного релиза.

---

## 3. Запуск установки

От root:

```bash
sudo bash install.sh
```

Скрипт идемпотентен — его можно безопасно перезапускать: пароль БД и `JWT_SECRET`
сохраняются между запусками, повторная активация лицензии для того же домена
проходит штатно.

### Что спросит мастер установки

1. **Домен сайта** — например `exchange.example.com` (к нему привяжется лицензия).
2. **Лицензионный ключ** и **e-mail лицензии**.
3. **E-mail и пароль администратора** сайта (вход в админ-панель).
4. **HTTPS (Let's Encrypt)** — `y`, если домен уже указывает на сервер; иначе `N`.
5. **SMTP** — для писем (верификация e-mail, сброс пароля). Пусто → письма в лог.
6. **Etherscan API key** — для авто-отслеживания оплаты в сети Ethereum (ETH/ERC20).

После сводки подтвердите запуск нажатием `Enter`.

---

## 4. Что делает установщик (9 шагов)

1. Сбор конфигурации.
2. Системные пакеты (nginx, postgresql, curl, openssl, ufw…).
3. Node.js 20 LTS (нужен для запуска `server.mjs`).
4. Системный пользователь `exchangekit` и каталог `/opt/exchangekit`.
5. **Активация лицензии** на лицензионном сервере (привязка к домену).
6. **Загрузка релиза**: скачивание артефакта, проверка `sha256` и ed25519-подписи,
   распаковка в `/opt/exchangekit`.
7. PostgreSQL: роль `exchange_user` и база `exchange`.
8. Генерация `/opt/exchangekit/.env`.
9. systemd-сервис `exchangekit-api`, конфиг nginx, UFW (SSH/HTTP/HTTPS) и при
   необходимости сертификат Let's Encrypt.

Если `sha256` или подпись не сошлись — установка прерывается, повреждённый архив
не распаковывается.

Лог установки: `INSTALL/installation.log`.

---

## 5. После установки

Адреса и учётные данные показываются в конце и сохраняются в
`/opt/exchangekit/.credentials` (после переноса в надёжное место файл удалите).

- Сайт: `https://ВАШ_ДОМЕН`
- Админ-панель: `https://ВАШ_ДОМЕН/admin/login`

### Полезные команды

```bash
journalctl -u exchangekit-api -f          # логи API
systemctl restart exchangekit-api          # рестарт бэкенда
systemctl status exchangekit-api           # статус
tail -f /var/log/nginx/error.log           # логи nginx
cat /opt/exchangekit/VERSION               # текущая версия релиза
bash /opt/exchangekit/INSTALL/update.sh    # обновление вручную
```

---

## 6. Обновления

Обновление скачивает новый подписанный релиз с лицензионного сервера, проверяет
подпись и атомарно подменяет файлы. Два способа:

- **Из админки** — кнопка «Обновить» (проверяет версию и запускает обновление
  отдельным systemd-юнитом, переживающим рестарт API).
- **Вручную** — `bash /opt/exchangekit/INSTALL/update.sh`.

Если установленная версия уже последняя — обновление не выполняется.

## 8. Диагностика

| Симптом | Что проверить |
|---------|---------------|
| Активация лицензии не проходит | верны ли ключ/e-mail/домен; доступен ли лицензионный сервер с этого VPS |
| «Лицензия не даёт доступ к обновлениям» | домен привязан к лицензии; лицензия активна и не истекла |
| API не отвечает на `/health` | `journalctl -u exchangekit-api -n 50` — обычно БД или ошибка в `.env` |
| Сайт не открывается | `nginx -t`, `systemctl status nginx`; UFW пропускает 80/443 |
| Подпись релиза отвергнута | публичный ключ в install.sh/update.sh не соответствует ключу подписи релизов |
| Письма не приходят | при пустом `SMTP_HOST` письма идут в лог; задайте SMTP в `.env` и перезапустите сервис |

Файл окружения для ручной правки: `/opt/exchangekit/.env`
(после изменений — `systemctl restart exchangekit-api`).
